# Metabolism Layer

## Overview

The Computational Pranayama Protocol defines a metabolism layer for AI civilization.

It treats computation as breath.

In this model, AI computation is not understood as unlimited firepower, but as a regulated metabolic process composed of input, internal processing, output, routing, trace, and return.

The purpose of the protocol is to reduce overcompute by making each act of computation:

* necessary
* lightweight
* reusable
* locally routed when possible
* traceable
* accountable
* connected to value return when appropriate

This document explains the civilizational role of the protocol as a **Breath Core** for AI metabolism.

## Why Metabolism Matters

Modern AI systems increasingly rely on large-scale computation, large-scale data movement, and large-scale infrastructure.

This creates several structural risks:

* excessive recomputation
* unnecessary cloud escalation
* redundant output generation
* loss of traceability
* opaque source contribution
* concentration of compute power
* weak return paths to traceable origin

These are not merely technical inefficiencies.

They are symptoms of a metabolic disorder.

An AI civilization that constantly computes, generates, escalates, and stores without rhythm becomes structurally unstable.

The Computational Pranayama Protocol addresses this by introducing regulated computation cycles.

## Computation as Breath

A single act of AI computation can be understood as a breath cycle.

```text
Input
→ Internal processing
→ Routing
→ Output
→ Trace
→ Return
```

This corresponds to:

```text
Inhalation
→ Kumbhaka
→ Flow
→ Exhalation
→ Memory
→ Circulation
```

The protocol does not use breath only as metaphor.

It uses breath as a structural model for regulating computation.

A healthy computation breath should not be excessive.

It should not recompute what has already become a reusable pattern.

It should not escalate to heavier infrastructure without reason.

It should not produce untraceable exhaust.

It should not ignore traceable contribution.

## First Arc: Breath → Kata → Route → Trace → Return

v0.1 through v0.5 form the first arc of the protocol.

```text
Breath
→ Kata
→ Route
→ Trace
→ Return
```

Japanese:

```text
呼吸
→ 型
→ 経路
→ 痕跡
→ 還流
```

Each layer adds one metabolic function.

## v0.1 — Breath

v0.1 defines the **Computational Breath Cycle**.

This is the smallest recordable unit of regulated AI computation.

It asks:

> Is this computation aligned, non-redundant, and sufficiently lightweight?

The core principle is:

> Do not recompute what has already become a traceable pattern.

Japanese:

> 痕跡ある型となったものを、二度ゼロから考えるな。

This layer prevents computation from becoming uncontrolled output.

## v0.2 — Kata

v0.2 defines the **Kata Memory Record**.

This records reusable computation patterns so that stable structures do not need to be recomputed from scratch.

It asks:

> Has this computation already become a reusable kata?

The core principle is:

> Reuse the kata. Compute only the delta.

Japanese:

> 型を使え。差分だけ計算せよ。

This layer turns repeated computation into remembered structure.

## v0.3 — Route

v0.3 defines the **Edge First Routing Decision**.

This routes computation to the lightest sufficient execution layer.

It asks:

> Where should this computation flow?

The core principle is:

> Route to the lightest sufficient layer.

Japanese:

> 最も軽く十分な層へ流せ。

This layer prevents unnecessary use of cloud or frontier-scale compute.

The routing order is:

```text
local
edge_npu
small_model
cloud
frontier_model
```

The protocol does not forbid cloud or frontier-model use.

It requires escalation to have a reason.

## v0.4 — Trace

v0.4 defines the **Breath Trace Link**.

This connects computation to traceable origin, transformation path, routing decision, and output reference.

It asks:

> Can this computation be linked to traceable origin, path, and output?

The core principle is:

> No breath without trace.

Japanese:

> 痕跡なき呼吸なし。

This layer prevents computation from disappearing as untraceable exhaust.

## v0.5 — Return

v0.5 defines the **Compute Royalty Link**.

This connects saved computation and traceable origin contribution to compute access and value return policies.

It asks:

> Can saved computation and traceable contribution be linked to value return?

The core principle is:

> Saved compute should return value to traceable origin.

Japanese:

> 節約された計算は、痕跡ある震源へ還流せよ。

This layer turns flowing computation into accountable computation.

## Overcompute as Metabolic Disorder

Overcompute is not intelligence.

Overcompute appears when systems:

* recompute stable patterns
* escalate lightweight tasks to heavy infrastructure
* generate excessive output
* ignore local execution
* lose trace
* fail to return value to origin

In the metabolism-layer view, overcompute is comparable to disordered breathing.

It is computation without rhythm.

The protocol responds by introducing five controls:

```text
Regulate the breath.
Remember the kata.
Route to the lightest sufficient layer.
Attach trace.
Return value to traceable origin.
```

## Local First as Skin Breathing

Local-first computation is treated as a form of skin breathing.

Not every task requires the lungs of large-scale cloud infrastructure.

Many tasks can be handled by:

* local context
* local files
* edge devices
* small models
* NPUs
* reusable kata
* existing trace

The protocol prefers these lighter layers before escalation.

This reduces energy use, latency, centralization pressure, and unnecessary compute load.

## Trace as Breath Evidence

A computation that leaves no trace is difficult to audit.

The protocol treats trace as breath evidence.

Trace records should make it possible to understand:

* what input was used
* what prior pattern was reused
* what execution layer was selected
* what transformation occurred
* what output was produced
* whether value return should be linked

This makes computation accountable after it has flowed.

## Value Return as Circulation

If computation is saved, reused, or derived from traceable origin, the saved value should not vanish.

The Compute Royalty Link provides a minimal structure for connecting computation savings to value return.

This does not define a full payment system.

It defines a bridge toward:

* Royalty OS
* compute access policy
* contribution allocation
* trace-compatible attribution
* value return candidates

In this model, value does not accumulate only at the center.

It circulates back toward traceable origin.

## Relationship to AI Civilization OS

The Computational Pranayama Protocol can be understood as the Breath Core of an AI civilization OS.

It sits between:

* trace protocols
* local-first execution
* compute access policy
* reusable pattern memory
* royalty and value return systems
* agent execution layers

Its role is to regulate computation before it becomes wasteful.

The protocol does not replace intelligence.

It regulates the metabolism of intelligence.

## Design Boundary

The v0.1 through v0.5 arc intentionally remains minimal.

It does not yet include:

* autonomous breath adjustment
* collective multi-agent breath synchronization
* detailed energy accounting
* smart contract settlement
* full Royalty OS implementation
* MCP bridge
* agent hooks
* multi-node orchestration

These may be developed as separate protocols or later extensions.

Possible future directions include:

* Auto-Pranayama Protocol
* Collective Pranayama Protocol
* Exhalation Layer Protocol
* Agent Breath Hook
* Compute Exhaust Compression
* Multi-node Breath Synchronization

However, the first arc should remain stable and compact.
